### 📄 ТЗ: Система визуального архива камер (Часть 1 из 3) — Архитектура, конфигурация, модель данных

> Это техническое задание описывает **уже реализованную систему**. Оно сознательно
> сформулировано независимо от конкретного языка/фреймворка: указанные технологии —

#### 1. Назначение системы 🎯

Менеджер визуального архива снапшотов и видеозаписей с камер видеонаблюдения,
лежащих на NAS/SMB-шаре. Приложение позволяет:

* быстро **ориентироваться** в больших архивах через тепловую карту (heatmap) с
  иерархическим погружением Год → Месяц → День → Час;
* **просматривать** кадры конкретного часа в виде хронологической ленты, с набором
  режимов визуализации движения и распознавания объектов (AI);
* **безопасно удалять** файлы с диска — с предпросмотром и автоматическим
  связыванием фото с соответствующим видео.

Интерфейс — тёмная панель в стиле умного дома (Home Assistant). Всё распознавание
объектов и генерация превью видео вынесены в отдельный необязательный сервис.

---

#### 2. Архитектура: три процесса + общий блок 🏗️

Система состоит из трёх запускаемых процессов и одного общего пакета кода:

| Блок | Референс | Роль | Состояние |
|---|---|---|---|
| **Frontend** | React + Vite (порт 5173) | UI: heatmap, просмотр часа, настройки | без состояния (всё в браузере) |
| **Main backend** | Python FastAPI (порт 8000) | БД, сканирование, лёгкие превью, motion-режимы, облачный AI, **все дисковые кэши**, маршрутизация к compute-service | владеет БД и кэшами |
| **Compute-service** | Python FastAPI (порт 8001) | локальная детекция объектов (YOLO/OpenVINO) и генерация превью видео | **без состояния** — принимает путь к файлу + параметры, возвращает результат |
| **Shared block** | общий пакет | контракт API (модели запрос/ответ) + словарь COCO→русский | импортируется обоими бэкендами |

**Ключевой принцип разделения:** main backend держит всё состояние (БД, кэши на
диске); compute-service не хранит ничего — он лишь выполняет тяжёлые вычисления по
запросу. Это позволяет вынести compute-service на отдельную машину или отключить
его полностью. Когда compute-service выключен, тяжёлые эндпоинты main backend
возвращают `503`, а фронтенд скрывает тяжёлые режимы (детекция, превью видео).

Поток в общем виде:

```
cameras.yaml
    │ (конфиг камер)
    ▼
config ──► scanner ──► database (SQLite)
                          ▲
   thumbnail pipeline ────┤   (basic / diff / diff_zoom / erosion / motion)
   cloud AI (gemini/claude)┤
   compute client ────────┘──HTTP──► compute-service (:8001)
                          │            YOLO-детекция · превью видео
                       HTTP API (FastAPI :8000)
                          │
                  фронтенд (Vite proxy /api → :8000)
                          │
              App ── HeatmapGrid / HeatmapCell      (тепловая карта)
                  └── HourViewer ── viewModes/       (просмотр часа)
                  └── ToolsModal                     (настройки, AI-ключи)
```

#### 3. Стек технологий (референс) 🛠️

Указано то, что используется в готовой реализации. Допустима замена на эквиваленты.

* **Frontend:** React (функциональные компоненты + хуки), сборка Vite. Графики —
  бесплатные open-source библиотеки (в референсе — Recharts для столбчатых
  диаграмм). Иконки — Material Design Icons (MDI), подключаются с CDN.
* **Main backend:** Python + FastAPI (асинхронный), сервер uvicorn. Обработка
  изображений — Pillow + numpy + OpenCV (cv2).
* **Compute-service:** Python + FastAPI. Детекция — Ultralytics YOLOv8, рантайм
  Intel OpenVINO (с откатом на PyTorch). Декодирование видео — OpenCV.
* **Облачный AI:** Google Gemini (SDK `google-genai`) и Anthropic Claude (SDK
  `anthropic`).
* **База данных:** SQLite — встроенная, файловая, без отдельного сервера. Схема
  плоская, с составными индексами под heatmap-запросы.
* **Конфигурация камер:** внешний YAML-файл.
* **Пользовательские настройки:** браузерный `localStorage` (на сервере не
  хранятся), с экспортом/импортом в YAML.

Запуск всех трёх процессов одной командой (в референсе — `npm start` через
`concurrently`); compute-service необязателен и может быть отключён или вынесен на
другую машину через настройку в UI (Tools → Compute).

#### 4. Доступ к хранилищу файлов 📁

* Бэкенд работает с путями к файлам как с **локальными директориями**. Монтирование
  сетевых дисков (SMB/UNC) и авторизация делегируются операционной системе, на
  которой запущен бэкенд.
* Поддерживаются как локальные пути (`C:\path\...`), так и UNC-шары
  (`\\server\share\...`).
* **Синхронизация только ручная.** Фонового мониторинга директорий нет. Обновление
  БД запускается исключительно по явному действию пользователя (кнопка
  «Пересканировать»). 🔄

---

#### 5. Конфигурация камер (YAML) ⚙️

Источники задаются внешним `.yaml`-файлом — добавление камеры не требует изменения
кода. Каждая камера имеет отдельные директории для снапшотов (фото) и видео.

```yaml
cameras:
  - id: "foscam_for_testing"
    name: "Foscam for testing"
    path_snapshots: "C:\\path\\to\\local\\snap"
    path_videos:    "C:\\path\\to\\local\\record"
  - id: "foscam_fi9805w"
    name: "Foscam FI9805W"
    path_snapshots: "\\\\192.168.1.99\\Camera\\Foscam\\snap"
    path_videos:    "\\\\192.168.1.99\\Camera\\Foscam\\record"
```

Поля камеры:

| Поле | Тип | Назначение |
|---|---|---|
| `id` | string | Уникальный идентификатор, ключ связи во всей системе и в БД |
| `name` | string | Человекочитаемое имя (отображается в UI) |
| `path_snapshots` | string | Директория с фото (локальный путь или UNC) |
| `path_videos` | string | Директория с видео |

Конфиг разбирается при старте бэкенда в структуру камеры (id, name,
path_snapshots, path_videos).

#### 6. Сканирование и разбор времени из имени файла 🔄

Эндпоинт сканирования (`POST /scan`) обходит директории камер из YAML, читает
системные атрибуты файлов (путь, размер, время) и обновляет БД. Логика:

* Можно сканировать одну камеру (`?camera_id=...`) или все сразу (без параметра).
* **Стратегия записи:** перед сканированием камеры все её записи удаляются из БД,
  затем создаются заново (полный `DELETE` + повторная вставка). Это гарантирует,
  что удалённые с диска файлы исчезают и из индекса.
* **Тип файла** (`photo` / `video`) определяется по расширению / директории.
* **Время кадра (`timestamp`)** извлекается из имени файла по известным шаблонам, а
  при отсутствии совпадения — из `mtime` файла (время модификации).

Поддерживаемые шаблоны имён (Foscam) — должны легко расширяться:

| Шаблон | Пример |
|---|---|
| Снапшоты Foscam | `MDAlarm_20231127-200442.jpg` |
| Видеозаписи Foscam | `alarm_20231127_200437.mkv` |

Точное совпадение времени снапшота и видео из этих шаблонов критично для функции
безопасного удаления (связывание фото ↔ видео по дельте времени, см. Часть 2).

#### 7. Структура базы данных (SQLite) 🗂️

Схема плоская, со строгими индексами — для мгновенной отрисовки heatmap без
задержек. Три таблицы.

##### 7.1. Таблица `files` — индекс файлов

Главная таблица. Одна строка на файл (фото или видео). Заполняется сканером.

| Колонка | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK | Автоинкремент |
| `camera_id` | TEXT | ID камеры из `cameras.yaml` |
| `file_type` | TEXT | `'photo'` или `'video'` |
| `file_path` | TEXT UNIQUE | Полный путь к файлу на диске |
| `file_size` | INTEGER | Размер в байтах |
| `timestamp` | TEXT | Время кадра в ISO-8601 (из имени файла или mtime) |

**Индексы (критично для скорости heatmap):**

* `idx_cam_ts` — `(camera_id, timestamp)`
* `idx_cam_type_ts` — `(camera_id, file_type, timestamp)`

##### 7.2. Таблица `thumbnails` — кэш базовых превью

Пути к сгенерированным базовым превью (256×256 JPEG). Генерируются лениво при
первом запросе превью.

| Колонка | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK | Автоинкремент |
| `file_id` | INTEGER UNIQUE | FK → `files.id` (**ON DELETE CASCADE**) |
| `thumb_path` | TEXT | Путь к файлу превью в кэш-директории |
| `created_at` | TEXT | Время создания (для автоочистки) |

Превью старше 30 дней удаляются автоматически; все превью можно очистить вручную
через maintenance-эндпоинт.

##### 7.3. Таблица `ai_analysis` — результаты AI-анализа

Результаты всех трёх провайдеров (Gemini, Claude, OpenVINO). Одна строка на файл.
Повторный запуск любого провайдера перезаписывает строку (UPSERT по `file_id`).

| Колонка | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK | Автоинкремент |
| `file_id` | INTEGER UNIQUE | FK → `files.id` (**ON DELETE CASCADE**) |
| `provider` | TEXT | `'gemini'` / `'claude'` / `'openvino'` |
| `model` | TEXT | Имя модели (напр. `gemini-2.5-flash`, `yolov8n`) |
| `analyzed_at` | TEXT | Время анализа |
| `scene_description` | TEXT | Тип сцены (улица, двор, парковка…). Пусто для OpenVINO |
| `image_description` | TEXT | Подробное описание кадра. Пусто для OpenVINO |
| `objects` | TEXT | Список ключевых слов объектов через пробел |

**Индекс:** `idx_ai_analysis_file` — `(file_id)`.
**Правило UPSERT:** `UNIQUE(file_id)` — последний запущенный анализ побеждает.

##### 7.4. Каскадные удаления

Удаление строки из `files` автоматически удаляет связанные строки в `thumbnails` и
`ai_analysis` (`ON DELETE CASCADE`):

```
files
  ├── thumbnails   (CASCADE DELETE)
  └── ai_analysis  (CASCADE DELETE)
```

БД создаётся автоматически при старте бэкенда. Вся БД может быть очищена вручную
(удаление всех записей `files`, без удаления файлов с диска) через
maintenance-эндпоинт.
