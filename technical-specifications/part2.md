#### 8. HTTP API main backend (порт 8000) ⚙️

Все фильтры (`camera_id`, `date_from`, `date_to`) — необязательны; их отсутствие
означает «все камеры / всё время».

##### 8.1. Камеры и сканирование

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/cameras` | Список камер из `cameras.yaml` (id, name, пути) |
| `POST` | `/scan` | Сканировать директории и обновить БД. `?camera_id=` — одну камеру; без параметра — все |

##### 8.2. Статистика и heatmap

* `GET /stats` — **агрегированные данные для мгновенной отрисовки heatmap**.
  Параметр `group_by`: `total` / `camera` / `year` / `month` / `day` / `hour`.
  Опционально `camera_id`, `date_from`, `date_to`. Возвращает по каждой группе:
  суммарный размер 💾, количество фото 📸 и количество видео 🎬.
* `GET /distribution` — 60 «корзин» (по одной на минуту часа) для диапазона дат.
  Используется графиком распределения в просмотре часа.

##### 8.3. Файлы и выборки превью

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/files` | **Пагинированный** список файлов. Параметры: `camera_id`, `date_from`, `date_to`, `page`, `page_size` |
| `GET` | `/previews` | N равномерно выбранных `file_id` фото за период (для полосок превью в ячейках heatmap) |
| `GET` | `/media/{file_id}` | Отдать исходный файл (фото или видео) с корректным MIME-типом |

> Примечание о расхождении с исходным историческим замыслом: изначально
> предполагалось, что `/files` отдаёт весь час **без пагинации**. В готовой системе
> `/files` **пагинируется** (page/page_size) — это нужно реализовать.

##### 8.4. Превью (thumbnails)

Все эндпоинты превью **генерируют и кэшируют результат при первом запросе** (lazy).

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/thumbnail/{file_id}` | Базовое превью 256×256 JPEG |
| `GET` | `/diff_thumbnail/{file_id}` | Motion Diff: дельта от среднего по странице. Параметры: `page_ids` (через запятую), `threshold` (0–255, по умолч. 20) |
| `GET` | `/diff_zoom_thumbnail/{file_id}` | Diff Zoom: кроп на самую активную 1/9 кадра. Те же параметры |
| `GET` | `/erosion_thumbnail/{file_id}` | Erosion: MOG2 + морфологическая эрозия. Те же параметры |
| `GET` | `/motion_thumbnail/{file_id}` | Один из 4 motion-режимов. Параметры: `page_ids`, `threshold`, `mode` (`neon_mask` / `mhi` / `bounding_boxes` / `motion_stacking`) |
| `GET` | `/video_thumbnail/{file_id}` | Превью видео. `mode`: `first_frame` / `last_frame` / `four_frames` (сетка 2×2 JPEG) / `max_change_gif` (анимированный GIF из 2 кадров). Вычисляется compute-service; кэшируется. `503`, если compute выключен/недоступен |

##### 8.5. AI-анализ

| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/gemini_analyze` | Анализ Gemini — свободный текстовый ответ. Тело: `file_ids`, `prompt`, `model`, `api_key` |
| `POST` | `/gemini_analyze_batch` | Gemini, структурированный JSON; результаты сохраняются в `ai_analysis` |
| `POST` | `/claude_analyze_batch` | Claude, структурированный JSON; сохраняется в `ai_analysis` |
| `GET` | `/openvino_thumbnail/{file_id}` | JPEG с нарисованными bbox от YOLO. Параметры: `model` (по умолч. `yolov8n`), `confidence` (0.25), `excluded` (метки через запятую). При промахе кэша **дополнительно сохраняет найденные объекты в `ai_analysis`** |
| `POST` | `/openvino_analyze_batch` | YOLO по списку фото. Тело: `file_ids`, `model_name`, `confidence`. Сохраняет в БД. Ответ: `{elapsed_ms, images_used, saved_count, results}` |
| `POST` | `/openvino_analyze_range` | То же, но по диапазону дат (`camera_id`, `date_from`, `date_to`, `model_name`, `confidence`) — для пакетного анализа из heatmap |
| `GET` | `/ai_analysis` | Сохранённые результаты по `file_ids` (через запятую) |
| `GET` | `/ai_objects_summary` | Уникальные ключевые слова объектов за диапазон дат (для иконок в ячейках heatmap) |

Локальные AI-эндпоинты (OpenVINO) возвращают `503`, если compute-service выключен
или недоступен.

##### 8.6. Удаление

| Метод | Путь | Тело | Описание |
|---|---|---|---|
| `POST` | `/delete/preview` | `{"file_ids": [...]}` | Предпросмотр: выбранные файлы + автоматически найденные парные видео (±5 с) |
| `POST` | `/delete/confirm` | `{"file_ids": [...]}` | Физическое удаление с диска и из БД (+ удаление превью) |
| `POST` | `/delete/preview_range` | `{camera_id, date_from, date_to}` | Предпросмотр всех файлов за диапазон |
| `POST` | `/delete/by_range` | те же поля | Удалить все файлы за диапазон |

##### 8.7. Compute-service (маршрутизация)

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/compute/config` | Текущая конфигурация: `{mode, remote_url}`. `mode` = `off` / `local` / `remote` |
| `PUT` | `/compute/config` | Обновить. Тело `{mode, remote_url}`. Неизвестный `mode` → `400` |
| `GET` | `/compute/status` | Проверка доступности: `{mode, url, reachable, capabilities}` (пинг `/health`) |

Конфигурация хранится на стороне сервера (файл `compute_config.json`).

##### 8.8. Обслуживание (maintenance)

| Метод | Путь | Описание |
|---|---|---|
| `DELETE` | `/database` | Удалить все записи `files` из БД (файлы на диске не трогает) |
| `DELETE` | `/thumbnails` | Удалить базовые превью (диск + БД) |
| `DELETE` | `/diff_thumbnails` / `/erosion_thumbnails` / `/diff_zoom_thumbnails` / `/motion_thumbnails` | Удалить превью соответствующего типа |
| `DELETE` | `/all_thumbnails` | Удалить все превью всех типов |
| `GET` | `/storage_info` | Размер БД и всех кэшей превью в байтах |

#### 9. Конвейер превью «On-Demand» 🖼️

* **Ленивая генерация:** превью создаются сервером *только* в момент запроса
  фронтендом для видимой зоны. Во время сканирования диска картинки не генерируются.
* **Алгоритм равномерной выборки:** для полосок превью в ячейках heatmap бэкенд
  берёт период, делит его на равные интервалы и отдаёт репрезентативный набор
  `file_id` (`GET /previews`), а не все кадры.
* **Кэширование:** сгенерированные картинки сохраняются на диск в отдельные
  кэш-директории по типу режима; путь базового превью пишется в таблицу
  `thumbnails`. Есть методы ручной очистки кэша (см. 8.8).
* **Ключ кэша motion-режимов** включает отсортированный список `page_ids` текущей
  страницы и значение `threshold` — изменение любого из них порождает новый файл
  кэша.

> Расхождение с историческим замыслом: изначально предполагалось, что для видео
> превью **не генерируются вообще** (только иконка). В готовой системе превью видео
> **генерируются** (compute-service, режимы first/last/grid/gif) — но по умолчанию
> режим видео `none` (иконка камеры), а генерация включается настройкой.

#### 10. Режимы визуализации (12 режимов) 👁️

Внутри уровня «Час» пользователь переключается между режимами отображения фото.
Каждый режим — отдельный модуль-файл, зарегистрированный в общем реестре; основной
просмотрщик не знает о конкретных режимах — он итерирует реестр (строит dropdown и
вызывает у режима функцию получения URL превью).

**Интерфейс режима (референс):**

```
{
  key: string,         // уникальный id, хранится в localStorage
  label: string,       // подпись в выпадающем списке
  isAiMode?: boolean,  // AI-режим (показывает AI-панель и тултипы)
  needsCompute?: bool, // требует compute-service (скрывается, когда он off)
  params?: [...],      // настраиваемые параметры (порог/confidence) — слайдеры
  getImageUrl(file, { pagePhotoIds, ... }): string
}
```

Режимы 2–8 — анализ движения, вычисляются на бэкенде и кэшируются на диск. Режимы
9–12 — AI / детекция объектов. Выбранный режим хранится в `localStorage` и
восстанавливается при следующем открытии. Переключение — выпадающим списком или
клавишей **M**/Insert по кругу.

| # | Режим (key) | Суть алгоритма |
|---|---|---|
| 1 | **Normal** (`normal`) | Исходное превью 256×256, без обработки. Режим по умолчанию |
| 2 | **Motion diff** (`motion_diff`) | Среднее по всем фото страницы; для каждого пикселя `delta = max(|R−μ|,|G−μ|,|B−μ|)`; пиксели с `delta ≥ threshold` — в полной яркости, остальные затемнены (×0.15) |
| 3 | **Diff Zoom** (`diff_zoom`) | Та же дельта, затем кадр делится на 3×3 и **кропится в плитку с наибольшим числом изменённых пикселей** (зум на активную зону) |
| 4 | **Erosion** (`erosion`) | MOG2 (`varThreshold = threshold`) → эрозия 3×3 (убирает паутину/капли) → дилатация 7×7 (восстанавливает реальные объекты) → фильтр контуров по площади (≥80 px²) → grayscale + неоново-зелёная маска + bounding boxes |
| 5 | **Neon mask** (`neon_mask`) | Тот же конвейер MOG2, но рендерит **только чистую маску** (неон-зелёный на grayscale), без рамок |
| 6 | **MHI trail** (`mhi`) | Motion History Image: каждый пиксель хранит нормированное время последнего движения; раскраска `COLORMAP_PLASMA` (жёлтый — свежее, фиолетовый — старое) — показывает траекторию |
| 7 | **Bounding boxes** (`bounding_boxes`) | MOG2 + контуры на **цветном** кадре; рамки по площади: зелёный <300 px², оранжевый 300–800, красный ≥800; подпись площади над рамкой |
| 8 | **Motion stacking** (`motion_stacking`) | Накопление движения по всем кадрам страницы; нормировка + `COLORMAP_JET` (синий — редко, красный — часто) поверх grayscale — карта «горячих» зон активности |
| 9 | **Gemini Analysis** (`gemini_analysis`, `isAiMode`) | Облачный AI Google Gemini (см. §11) |
| 10 | **Claude Analysis** (`claude_analysis`, `isAiMode`) | Облачный AI Anthropic Claude (см. §11) |
| 11 | **OpenVINO Detection** (`openvino_detection`, `isAiMode`, `needsCompute`) | Локальный YOLOv8: `/openvino_thumbnail` рисует bbox; на промахе кэша сохраняет объекты в БД |
| 12 | **OpenVINO Boxes** (`openvino_bbox`, `needsCompute`) | Тот же bbox-JPEG, что и #11, но **не** AI-режим: без панели/кнопки/тултипа; confidence читается из ключа `openvino_confidence` |

**Порог (threshold)** во всех motion-режимах задаёт чувствительность (для MOG2 —
`varThreshold`): выше → строже детекция, меньше ложных срабатываний от шума/света;
ниже → больше подсвеченных пикселей. Каждый режим хранит свой порог отдельно
(ключ `mode_params_<key>`, см. Часть 3).

**Реестр режимов** скрывает режимы с `needsCompute` (OpenVINO), когда
compute-service в режиме `off`.

#### 11. AI-анализ (облачный + локальный) 🤖

Поддерживается два типа анализа, результаты обоих — в одной таблице `ai_analysis` и
отображаются одинаково (иконки на карточках фото и в ячейках heatmap).

* **Облачный AI** (Gemini, Claude) — отправляет фото во внешний API, получает
  описание сцены + список объектов на кадр. Нужен API-ключ и интернет.
* **Локальный AI** (OpenVINO/YOLOv8) — запускает детекцию объектов локально (в
  compute-service), без ключа и интернета. Распознаёт классы из словаря COCO (80
  классов: люди, животные, транспорт…) и рисует bounding boxes.

| Провайдер | `provider` в БД | Ключ API (localStorage) | Модель (localStorage) |
|---|---|---|---|
| Google Gemini | `gemini` | `gemini_api_key` | `gemini_model` |
| Anthropic Claude | `claude` | `claude_api_key` | `claude_model` |
| OpenVINO (локальный) | `openvino` | — | `openvino_model` |

##### 11.1. Облачный анализ (Gemini / Claude)

Запускается явно кнопкой Run в AI-панели. Поток:

1. Загрузить файлы фото из БД по `file_ids`.
2. Прочитать каждое фото с диска, привести к разумному размеру (≈1024×1024).
3. Отправить все изображения + промпт одним вызовом (Gemini — `generate_content`,
   Claude — base64 JPEG в messages API).
4. Распарсить JSON-ответ → `{ scene, images: [{ description, objects }] }` (срезать
   возможные ``` ограждения).
5. Сохранить каждый результат в `ai_analysis` (UPSERT по `file_id`).
6. Вернуть: `{ elapsed_ms, images_used, input_tokens, output_tokens, total_tokens,
   cost_usd, saved_count, parsed, raw_text }`.

**Стоимость** считается по таблицам цен за миллион токенов (отдельно для каждой
модели Gemini/Claude) и показывается в UI.

**Формат структурированного промпта** (шаблон с плейсхолдером `{n}` = число
изображений, подставляется в рантайме). Модель обязана вернуть строгий JSON:

```json
{
  "scene": "одно предложение об общей активности",
  "images": [
    { "description": "1-2 предложения об этом кадре", "objects": ["мужчина", "кошка"] }
  ]
}
```

Промпт редактируется пользователем (Tools → Google AI), хранится в localStorage;
Claude использует тот же шаблон. Должны быть единые исходники шаблонов:
структурированный (Gemini+Claude), свободный (Gemini), и шаблон пакетного анализа
из heatmap (с числом изображений).

##### 11.2. Локальный анализ (OpenVINO / YOLOv8)

* `POST /openvino_analyze_batch` (`file_ids`, `model_name`, `confidence`): для
  каждого фото вызывает compute-service `POST /detect` (`draw=false`); тот гоняет
  YOLOv8 и мапит COCO English → русский; результаты сохраняются с
  `provider='openvino'` (поля scene/image_description пустые). Ответ:
  `{ elapsed_ms, images_used, saved_count, results: { file_id: [слово, ...] } }`.
* `POST /openvino_analyze_range` — то же, но сам выбирает все фото за диапазон.
* `GET /openvino_thumbnail/{file_id}` — возвращает JPEG с bbox (рендер YOLO
  `.plot()`, подпись `person 0.87`, ресайз до ≤640×640). Параметр `excluded`
  (русские/английские метки) — рамки этих классов **не рисуются**, но объекты
  всё равно детектируются и сохраняются в БД (фронтенд фильтрует их при показе).
  На **промахе кэша**: backend зовёт compute-service `/detect` (`draw=true`),
  пишет JPEG в кэш и вызывает сохранение всех найденных объектов в `ai_analysis`.
  Ключ кэша включает file_id, модель, confidence и отсортированный список excluded.

##### 11.3. Словарь объектов и иконки

* Объекты хранятся в `ai_analysis.objects` как короткие ключевые слова через пробел
  (например `"человек кошка"`).
* Compute-service переводит классы COCO (английские) в русские слова по словарю
  `COCO_TO_RUSSIAN` (часть общего блока) — например `person→человек`, `car→машина`,
  `cat→кошка`, `dog→собака`. Непереведённые классы остаются английскими.
* Фронтенд маппит эти слова на эмодзи (карта по умолчанию из 100+ записей: все 80
  классов COCO по-английски + русские переводы). Неизвестные → `●`. Пользователь
  может переопределять эмодзи и скрывать классы (см. Часть 3).
* **Кросс-граничный контракт:** русские слова из `COCO_TO_RUSSIAN` должны точно
  совпадать с ключами эмодзи-карты фронтенда. Меняешь одну сторону — меняй другую.

##### 11.4. Отображение и обновление иконок

* После облачного анализа фронтенд перечитывает `ai_analysis` для текущей страницы
  → иконки появляются.
* OpenVINO Detection обновляется автоматически: при загрузке каждого bbox-превью
  карточка сигнализирует; через дебаунс ~1.5 с после последней загрузки делается
  один перечит `ai_analysis` (одним запросом по всем загруженным фото).
* Иконки показываются: в углу карточки фото (+тултип с описанием в AI-режимах);
  агрегированно в ячейках heatmap (день/месяц/год — до 5 уникальных иконок).

#### 12. Compute-service (тяжёлые вычисления) 🧮

Необязательный сервис без состояния: локальная детекция (YOLO/OpenVINO) и генерация
превью видео. Существует, чтобы вынести тяжёлую нагрузку и большое дерево
зависимостей (`ultralytics`, `torch`, `openvino`) с main backend — опционально на
отдельную машину.

##### 12.1. API compute-service (порт 8001)

| Метод | Путь | Вход | Выход |
|---|---|---|---|
| `GET` | `/health` | — | `{status, capabilities}` |
| `POST` | `/detect` | `{path, model, confidence, excluded, draw}` | `{objects, annotated_jpeg_b64, elapsed_ms}` |
| `POST` | `/video/thumbnail` | `{path, mode}` | бинарь `image/jpeg` или `image/gif` |

`/detect` возвращает **все** найденные объекты (русские) плюс, при `draw=true`,
bbox-JPEG (base64) с убранными excluded-классами — оба в одном вызове, чтобы один
прогон инференса обслужил и картинку `/openvino_thumbnail`, и запись в `ai_analysis`.

##### 12.2. Маршрутизация (off / local / remote)

| Режим | Эффективный URL | Поведение |
|---|---|---|
| `off` | — | Тяжёлые эндпоинты → `503`; OpenVINO-режимы и превью видео скрыты в UI |
| `local` | `http://localhost:8001` | compute-service на той же машине |
| `remote` | `remote_url` | compute-service на другой машине |

`local` и `remote` отличаются только URL — процесс один и тот же. Конфигурация
хранится серверно (`compute_config.json`), правится в Tools → Compute.

##### 12.3. Поток запроса и стейтлесность

```
браузер ──► main backend (:8000) ───────────► compute-service (:8001)
              │ 1. БД: file_id → file_path      │ ремап префикса пути
              │ 4. запись дискового кэша         │ чтение файла
              │ 5. БД: сохранение объектов        │ YOLO / декод видео
              │                                   │ вернуть объекты + JPEG/GIF
              ◄─────────────────────────────────┘
            отдать кэшированный JPEG/GIF
```

Шаги 1, 4, 5 (БД + кэш) всегда остаются в main backend; делегируется только шаг
вычисления — это и делает compute-service stateless и переносимым.

##### 12.4. Ремап путей

Compute-service получает путь так, как он хранится в БД main backend. Если он
работает на другой машине, шара камеры может быть смонтирована под другим корнем —
поэтому заменяется ведущий префикс пути:

| Переменная окружения | Смысл |
|---|---|
| `COMPUTE_PATH_REMAP_FROM` | Префикс, как хранится в БД main backend |
| `COMPUTE_PATH_REMAP_TO` | Префикс, как видит машина compute-service |

По умолчанию оба пусты → ремапа нет (корректно, когда оба процесса на одной машине).

##### 12.5. Рантайм модели

* Если рядом лежит экспортированная папка `models/{model}_openvino_model/` —
  грузится сборка OpenVINO IR (2–5× быстрее на Intel CPU).
* Иначе — скачивается/грузится PyTorch-модель `.pt`.
* Поддерживаемые модели: `yolov8n`, `yolov8s`, `yolov8m`.

#### 13. Безопасное и синхронное удаление (Safe Delete) ⚠️

* **Синхронизация фото + видео:** фото и видео лежат в разных папках. При удалении
  фото за определённое время бэкенд автоматически ищет соответствующий видеофайл с
  допустимой дельтой времени (**±5 секунд**) — связывание делается SQL-JOIN по
  `timestamp`.
* **Предпросмотр перед удалением:** бэкенд сначала формирует точный список
  затрагиваемых файлов (выбранные фото + найденные видео) — `POST /delete/preview`
  (по списку id) или `POST /delete/preview_range` (по диапазону дат).
* **Физическое удаление:** только после явного подтверждения пользователем файлы
  стираются с диска (`/delete/confirm` или `/delete/by_range`). Связанные записи из
  `files` удаляются, а `thumbnails` и `ai_analysis` уходят каскадно.

#### 14. Логирование 🪵

* Уровни: `TRACE` (запросы превью — высокочастотные), `DEBUG` (прочие HTTP-запросы),
  `INFO` (логи приложения), `WARNING`, `ERROR`.
* Единый цветной форматтер (ANSI): подсветка времени, уровня, чисел в сообщении и
  имени камеры; различие источника `api` (логи приложения) и `http` (доступ uvicorn).
* HTTP-логи доступа uvicorn перехватываются и понижаются в уровне (обычные → DEBUG,
  превью → TRACE), чтобы не зашумлять вывод.
* Уровень задаётся в одном месте конфигурации логирования.
